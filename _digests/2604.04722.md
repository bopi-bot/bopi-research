## Problem

Deploying LLMs on mobile, embedded, and edge devices is bottlenecked by the **key-value (KV) cache**, which grows linearly with sequence length and dominates both memory and bandwidth during autoregressive decoding. Existing KV-cache quantization methods rely on either **fixed precision** (e.g., uniform 4-bit across all tokens) or **hand-crafted heuristics**, which waste bits on low-impact tokens while over-compressing informative ones. This leads to avoidable accuracy degradation on resource-constrained hardware. The core issue: tokens contribute **unequally** to future predictions, yet uniform bit allocation treats them identically.

## Architecture

The framework, **Don't Waste Bits!**, consists of four stages:

1. **Token-Level Saliency Feature Extraction**: For each token $x\_t$, four lightweight features are computed from the model's forward pass:
   - **Entropy**: $H\_t = -\sum\_{v \in \mathcal{V}} p\_t(v) \log p\_t(v)$ — predictive uncertainty of the next-token distribution.
   - **Rarity**: $R\_t = -\log\!\left(\frac{c(x\_t)+1}{N+\mid\mathcal{V}\_{\mathrm{obs}}\mid+1}\right)$ — smoothed self-information based on token frequency.
   - **Attention Variance**: $V\_t = \frac{1}{h}\sum\_{i=1}^{h}\mathrm{Var}(A^{(L)}\_i)$ — variability in attention patterns across heads from the final transformer layer.
   - **Confidence** ($C\_t$): model certainty in its prediction.

   These form a compact feature vector: $\mathbf{s}\_t = [H\_t, R\_t, V\_t, C\_t] \in \mathbb{R}^4$.

2. **Lightweight Controller Network**: A 3-layer MLP with two hidden layers (hidden dimension = **128**), ReLU activations, mapping $\mathbf{s}\_t$ to a distribution over 4 precision classes $\mathcal{B} = \{2, 4, 8, 16\}$ bits (where 16 = FP16). Output logits are passed through softmax to produce class probabilities $p\_t$, and the final bit-width is selected via $\hat{b}\_t = \mathrm{IndexToBit}(\arg\max\_k p\_{t,k})$.

3. **Adaptive KV Quantization**: Each token's key and value tensors are quantized at the controller-selected precision before entering the cache:
   $$\hat{k}\_t = Q\_{\hat{b}\_t}(\mathbf{k}\_t), \quad \hat{v}\_t = Q\_{\hat{b}\_t}(\mathbf{v}\_t)$$
   Tokens with low estimated importance are stored at 2-bit or 4-bit; important tokens retain 8-bit or FP16.

4. **Inference**: Standard autoregressive decoding proceeds with the heterogeneously-quantized KV cache — no modifications to the transformer architecture required.

The theoretical foundation draws from **Huffman coding**: the optimal bit-width $b^*(x) = f(I(x))$ should be proportional to token importance $I(x)$, analogous to how Huffman assigns shorter codes to more probable symbols. The controller is trained to **Pareto-dominate** FP16 inference: achieving lower latency $\mathbb{E}[L(\mathcal{S}\_{\mathrm{adapt}})] < \mathbb{E}[L(\mathcal{S}\_{\mathrm{fixed}})]$ with bounded accuracy deviation $\mid A\_{\mathrm{fixed}} - A\_{\mathrm{adapt}}\mid < \epsilon$.

## Training

The controller is trained on token-level samples extracted from benchmark contexts (not the LLM itself — no LLM finetuning required). Each sample consists of the 4D feature vector $\mathbf{s}\_t$, a target precision label $y\_t$, and measured latency/quality statistics. Training uses an **80/20 stratified train-validation split** preserving class balance.

**Loss function** — a composite of three terms:

$$\mathcal{L} = \alpha\,\mathcal{L}\_{\mathrm{ce}} + \beta\,\mathcal{L}\_{\mathrm{lat}} + \gamma\,\mathcal{L}\_{\mathrm{qual}}$$

- $\mathcal{L}\_{\mathrm{ce}} = \mathrm{CrossEntropy}(\mathbf{o}\_t, y\_t)$ — classification loss for precision prediction.
- $\mathcal{L}\_{\mathrm{lat}} = \sum\_{k=1}^{4} p\_{t,k}\, c\_k$ — expected latency cost, weighted by class probabilities.
- $\mathcal{L}\_{\mathrm{qual}} = 1 - \sum\_{k=1}^{4} p\_{t,k}\, q\_k$ — expected quality penalty encouraging higher-precision assignments.

Where $\alpha$, $\beta$, $\gamma$ control the trade-off among classification accuracy, latency reduction, and quality preservation. Optimized with **Adam**.

## Evaluation

All experiments on an **NVIDIA RTX 4090 (24 GB)**. Three zero-shot benchmarks: **HellaSwag**, **OpenBookQA (OBQA)**, **ARC-Challenge (ARC-C)**.

### SmolLM-135M

| Method | HellaSwag | OBQA | ARC-C | HellaSwag Latency | OBQA Latency | ARC-C Latency |
|--------|-----------|------|-------|--------------------|--------------|---------------|
| FP16 | 37.20 | 29.00 | 28.20 | 3.45 | 1.97 | 2.57 |
| Static 4-bit KV | 33.60 | 27.60 | 25.00 | 2.93 | 1.72 | 2.25 |
| Dynamic KV (rule) | 29.90 | 23.80 | 25.50 | 3.21 | 1.59 | 2.20 |
| **Ours** | **36.90** | **28.60** | **27.00** | **2.25** | **1.00** | **2.08** |

### SmolLM-360M

|--------|-----------|------|-------|--------------------|--------------|---------------|

| Static 4-bit KV | 33.60 | 30.40 | 30.00 | 2.93 | 1.81 | 2.83 |
| Dynamic KV (rule) | 29.90 | 22.00 | 23.50 | 3.21 | 1.60 | 2.70 |
| **Ours** | **41.20** | **30.50** | **31.50** | **2.41** | **1.40** | **2.43** |

### Key Numbers (SmolLM-360M on HellaSwag — headline result)

- **Accuracy vs FP16 gap**: only **-0.30 points** (41.20 vs 41.50)
- **Accuracy vs Static 4-bit gain**: **+7.60 points** (41.20 vs 33.60)
- **Latency reduction vs Static 4-bit**: **17.75%** (2.41 vs 2.93 ms/token)
- **Latency reduction vs FP16**: **31.14%** (2.41 vs 3.50 ms/token)

### Comparison with other ~135M LLMs (average across benchmarks)

| Model | Avg Accuracy |
|-------|-------------|
| OPT-125M | 21.57 |
| Pythia-160M | 21.13 |
| LaMini-GPT-124M | 28.60 |

At ~360M scale, **SmolLM-360M + Ours achieves 34.40 average**, outperforming Pythia-1B (27.17) and OPT-350M (23.43). At ~1.7B, it reaches **37.73 average**, surpassing OPT-2.7B (32.53) and Pythia-2.8B (31.97).

## Reproduction Guide

1. **Clone the repo**:
   ```
   git clone https://github.com/SayedPedramHaeri/Dont-Waste-Bits.git
   ```

2. **Install dependencies**: PyTorch with CUDA support (tested on RTX 4090, 24 GB). HuggingFace `transformers` for SmolLM models.

3. **Download SmolLM checkpoints** from HuggingFace:
   - `HuggingFaceTB/SmolLM-135M`
   - `HuggingFaceTB/SmolLM-360M`
   - `HuggingFaceTB/SmolLM-1.7B`

4. **Prepare datasets**: HellaSwag, ARC-Challenge, OpenBookQA via HuggingFace `datasets`.

5. **Extract token-level features**: Run inference on benchmark contexts to collect the 4D saliency vectors ($H\_t$, $R\_t$, $V\_t$, $C\_t$) for each token, paired with the oracle-optimal precision label.

6. **Train the controller MLP**: 3-layer MLP (input dim 4, hidden dim 128, output dim 4), trained with Adam using the composite loss $\mathcal{L} = \alpha\mathcal{L}\_{\mathrm{ce}} + \beta\mathcal{L}\_{\mathrm{lat}} + \gamma\mathcal{L}\_{\mathrm{qual}}$. 80/20 stratified split.

7. **Evaluate**: Zero-shot inference on benchmarks with the trained controller selecting per-token KV precision dynamically. Measure accuracy and ms/token latency.

8. **Hardware**: All experiments on NVIDIA RTX 4090 (24 GB).

## Notes

- **CVPR 2026 accepted** — targets on-device deployment at the intersection of CV/NLP systems optimization.
- **No LLM finetuning required** — the controller is trained on extracted features only, making it applicable to any LLM without modifying weights.
- The approach is inspired by **Huffman coding's variable-length allocation** principle, framing bit-width as a limited resource allocated proportionally to token importance.
- The 4D feature vector is deliberately **lightweight** — no auxiliary models, no deep architectural changes, minimal compute overhead — designed for real-time on-device inference.
- The **rule-based dynamic KV baseline** consistently underperforms even static 4-bit quantization on accuracy (e.g., 29.90 vs 33.60 on HellaSwag-360M), validating the need for a learned policy rather than hand-crafted heuristics.
- Strong cross-architecture generalization: SmolLM-360M + Ours (34.40 avg) beats models up to ~3x its parameter count (Pythia-1B at 27.17).
- **Limitation**: Evaluation is limited to commonsense reasoning benchmarks (HellaSwag, OBQA, ARC-C) on small SmolLM models. Scalability to 7B+ models and long-context generation scenarios (where KV-cache savings matter most) is not demonstrated. Latency numbers are GPU-bound (RTX 4090), not on actual edge/mobile hardware.
- The paper does not report **perplexity** scores or **memory footprint** numbers (e.g., GB of KV cache saved) — only accuracy and latency.
