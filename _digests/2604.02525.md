## Problem

Low-precision training (LPT) for LLMs commonly applies Hadamard transforms to suppress outliers before quantization. Prior methods (MXFP4+Hadamard, HALO, Quartet, HOT) all apply a **fixed** Hadamard transform uniformly across all layers and computation paths. This paper shows that strategy is **fundamentally suboptimal**: the effectiveness of Hadamard-based suppression depends critically on how the transform's smoothing direction aligns with the outlier structure of each operand pair — a property that varies substantially across layers and paths.

**This is the first systematic study** of outlier patterns across **weights, activations, and gradients** of LLMs. Three distinct outlier pattern types are identified:

- **Row-wise (R)**: Outliers concentrated in a small number of rows (e.g., gradient tensors — certain token positions have consistently larger gradient magnitudes)
- **Column-wise (C)**: Outliers concentrated in a small number of columns (e.g., activation tensors — certain feature dimensions exhibit consistently larger magnitudes)
- **None (N)**: Relatively uniform magnitude distribution (e.g., weight tensors)

These patterns yield 9 possible pattern pairs for each matrix multiplication. The paper shows that **IHT (Inner Hadamard Transform) is only effective for CR pairs** — where its smoothing directions (column of left operand, row of right operand) are orthogonal to both outlier directions. For pairs like RC, RN, RR, NC, and CC, IHT is ineffective or even harmful (can increase quantization error).

Key empirical finding: outlier patterns are **stable throughout training** (tracked over 300 steps on Llama3.2-3B). This makes one-time calibration feasible.

## Architecture

AdaHOP (**Ada**ptive **H**adamard transform with **O**utlier-**P**attern-aware strategy) assigns each matrix multiplication its optimal strategy based on the detected outlier pattern pair. The framework has two variants:

**Calibration phase (one-time, before LPT):**
1. Run BF16 training for 30 steps
2. Compute dimension-normalized Coefficient of Variation (CV) along row and column dimensions for each tensor
3. Classify each tensor as R, C, or N using threshold $\tau = 2.0$
4. Fix the pattern per tensor by majority vote over calibration steps

This costs $< 0.01\%$ of total training budget (20B–160B tokens).

**Strategy selection per pattern pair (Table 2 in paper):**

| Left ($A$) | Right ($B$) | Strategy |
|:---:|:---:|:---|
| C | N | IHT only |
| N | N | IHT only |
| R | N | OE-Left + IHT |
| R | C | OE-Right + IHT |
| N | C | OE-Right + IHT |
| C | C | OE-Right (Lv1) / BF16 (Lv2) |

**OE-Left**: Decompose $A = A\_{\text{res}} + A\_{\text{out}}$ where $A\_{\text{out}}$ contains the top-$k$ outlier rows ($k = 64$). Compute $C = Q(A\_{\text{res}} H\_k) \cdot Q(H\_k^\top B) + A\_{\text{out}} \cdot B$ (MXFP4 + BF16).

**OE-Right**: Decompose $B = B\_{\text{res}} + B\_{\text{out}}$ where $B\_{\text{out}}$ contains top-$k$ outlier columns. Compute $C = Q(AH\_k) \cdot Q(H\_k^\top B\_{\text{res}}) + A \cdot B\_{\text{out}}$ (MXFP4 + BF16).

**CC pairs** account for $< 6\%$ of all computation paths but appear primarily in Key/Value projection layers (attention-critical):
- **AdaHOP-Lv1**: OE-Right with IHT on residual
- **AdaHOP-Lv2**: Full BF16 for CC pairs

**Hardware-aware Triton kernels** on AMD CDNA4 (MI350 series) with 4 pipeline stages:
1. Fast Outlier Index Detection (FOID) — top-$k$ by row/column variance
2. Fused IHT + MXFP4 quantization (single kernel, no intermediate writes)
3. Fused MXFP4+BF16 GEMM (concurrent via CDNA4 mixed-precision compute units)
4. Fused Scatter-Add for in-place accumulation

$k = 64$ is hardware-motivated: CDNA4's `mfma_scale_f32_32x32x64_f8f6f4` instruction consumes a $32 \times 64$ input tile, so the BF16 outlier sub-matrix fits a minimal $2 \times 1$ MFMA tile arrangement.

## Training

**Models tested:**
- Llama3.2-1B, Llama3.2-3B, Instella-3B, Llama3.1-8B

**Dataset:** C4 (Colossal Clean Crawled Corpus)

**Precision format:** MXFP4 (4-bit floating-point with shared scaling)

**Optimizer:** AdamW with $\text{lr} = 4 \times 10^{-4}$, $\epsilon = 10^{-8}$

**Schedule:** Linear warmup over 200 steps

**Batch size:** Global batch size of 128, sequence length 4,096 tokens

**Training tokens (Chinchilla-optimal scaling):**
- Llama3.2-1B: 40B tokens
- Llama3.2-3B, Instella-3B: 60B tokens each
- Llama3.1-8B: 160B tokens

**Gradient clipping:** Max norm 1.0

**Hardware:** AMD CDNA4 (MI350 Series GPU), Triton kernels

**Baselines:** BF16, naive MXFP4, MXFP4+Hadamard (uniform IHT), Tseng et al. (randomized Hadamard), HALO (OHT on gradient paths)

- **AdaHOP-Lv1**: 3.6$\times$ memory compression, ~2% faster than BF16 end-to-end
- **AdaHOP-Lv2**: 2.7$\times$ memory compression, ~1% faster than BF16
- **HALO's throughput is catastrophically low** (10,482 tok/s, below BF16) due to OHT requiring two extra FWHT kernel invocations per activation gradient path
- Kernel-level GEMM speedup: **1.59–1.80$\times$** over BF16 (end-to-end is less due to non-GEMM ops)

### Kernel Latency Breakdown (Llama3.1-8B)

For $2048 \times 14336$ (MLP projection): total 0.297ms vs 0.473ms BF16 = **1.59$\times$ speedup**
For $4096 \times 4096$ (attention projection): total 0.193ms vs 0.347ms BF16 = **1.80$\times$ speedup**

Overhead breakdown (MLP path): FOID 0.035ms + IHT+Quant 0.052ms + Fused GEMM 0.192ms + Scatter-Add 0.017ms = 0.297ms

## Reproduction Guide

1. **Environment**: AMD CDNA4 GPU (MI350 series), Triton, PyTorch with MXFP4 support
2. **Obtain model**: Use HuggingFace Llama3.2-1B/3B, Llama3.1-8B, or Instella-3B
3. **Dataset**: Download C4 dataset, configure sequence length 4096, global batch size 128
4. **Calibration**: Run 30 BF16 training steps. At each step, compute row-wise and column-wise CV for every tensor in every linear layer. Classify as R/C/N using $\tau = 2.0$. Fix patterns by majority vote
5. **Strategy assignment**: For each linear layer's 3 matmuls (forward, $\nabla W$, $\nabla X$), determine pattern pair from operand patterns and assign strategy per Table 2
6. **Training**: Configure AdamW (lr=$4 \times 10^{-4}$, $\epsilon=10^{-8}$), linear warmup 200 steps, grad clip 1.0. Switch to MXFP4 with AdaHOP strategy. Train for Chinchilla-optimal tokens (40B/60B/160B)
7. **Evaluation**: Zero-shot on PIQA, HellaSwag, ARC-Easy, LAMBADA
8. **No public code available** as of paper publication (April 2026)

## Notes

**Connection to hadamard-rotation-quantization-dominance.md:** That note observes Hadamard rotation accounts for 98% of quantization improvement at 5-bit — rotation is the dominant factor over centroid design. AdaHOP **refines this picture**: Hadamard is indeed the best rotation on average, but it is **not optimal for every tensor**. For 6 of the 9 possible outlier pattern pairs (RN, RC, RR, NC, CC, NN), uniform IHT is ineffective or harmful. The paper shows IHT only helps for CN and CR pairs in practice.

**Key nuance:** The paper challenges the "one-size-fits-all" assumption shared by MXFP4+Hadamard, HALO, Quartet, and HOT. Even HALO — which correctly identifies that gradients have row-wise outliers and activations have column-wise outliers — still applies rotations uniformly regardless of the actual pattern pair at each matmul. AdaHOP's insight is that each matmul has its own optimal strategy.

**Practical implications for edge deployment:**
- The calibration overhead is negligible ($< 0.01\%$ of training), making this practical even for resource-constrained settings
- The pattern stability finding means no runtime detection needed — a static lookup table suffices
- AdaHOP-Lv1 achieves BF16 quality at MXFP4 precision with only 3.6$\times$ memory compression — relevant for edge training scenarios
- The $k = 64$ extraction parameter is hardware-tied to CDNA4's MFMA tile size; on different hardware, this would need retuning

**Limitations:** Only tested on Llama-family and Instella architectures (1B–8B). Weight tensors always showed None pattern — may differ for other architectures (Mixtral MoE, Gemma). Fixed $k = 64$ throughout — adaptive per-layer $k$ is future work. No public code available.
