## problem

binary quantization reduces LLM weights to $\pm 1$ for maximal memory savings (32$\times$ compression vs FP16), but existing approaches to push below 1 bit per weight rely on sparsity masks (e.g., STBLLM), which create three problems: (1) accuracy degrades sharply when pruning binary weights, (2) sparse mask management adds 0.25 bits/weight overhead with semi-structured pruning, and (3) irregular sparsity patterns have poor hardware compatibility outside specialized accelerators. prior 1-bit PTQ methods like BiLLM (Huang et al., 2024) use residual binarization to preserve salient weights, and ARB-LLM (Li et al., 2025) iteratively refines bias and scaling factors. both achieve 1.11 bits but cannot go lower without quality collapse. the open question: how to achieve sub-1-bit compression for binary LLM weights without sparse masks, while maintaining performance on standard hardware.

## architecture

BTC-LLM has two components: a **binary codebook** that exploits redundancy in binarized weight vectors, and a **learnable transformation** ($T = D\_{\pm} P$) that reduces activation outliers and promotes shared sign patterns.

**binary codebook.** binary weight matrices are split into sub-vectors of length $v$, each mapping to one of $c$ centroid vectors in $\{\pm 1\}^v$. the total storage per weight is $\lceil \log\_2 c \rceil / v$ bits plus the negligible codebook overhead ($v \cdot c$ bits). for $v = 128$ and $c = 512$, this yields 0.8125 bits/weight. the codebook is trained with a modified K-means:

1. **initialization**: extract unique binary vectors $\mathcal{U}$ from the weight matrix; if $\mid \mathcal{U} \mid \geq K$, pick the $K$ most frequent as centroids, otherwise use all unique vectors.
2. **E-step**: assign each binary vector to nearest centroid via Hamming distance $d\_H(\mathbf{b}, \mathbf{c}) = \text{POPCNT}(\mathbf{b} \oplus \mathbf{c})$, implemented with XOR + POPCNT on packed int64.
3. **M-step**: update centroid as $\mathbf{c}\_k = \text{sign}\big(\frac{1}{\mid \mathcal{B}\_k \mid} \sum_{\mathbf{b}\_i \in \mathcal{B}\_k} \mathbf{b}\_i\big)$ with $\text{sign}(0) = +1$.

inference uses **LUT-GEMM**: pre-compute dot products between input activation blocks and all codebook centroids, store in a lookup table, then replace the GEMM with table lookups and accumulation. this avoids runtime dequantization entirely.

**learnable transformation.** $D\_{\pm} = \text{diag}(\sigma)$ where $\sigma\_i \in \{\pm 1\}$ is a diagonal sign-flip matrix (invertible, no magnitude change). $P = P\_1 \otimes P\_2$ is a learnable invertible affine matrix parameterized as a Kronecker product (following FlatQuant, Sun et al., 2024). $P$ is optimized per transformer block via:

$$\min\_{T\_l} \big(\| \mathcal{F}\_l(X) - \hat{\mathcal{F}}\_l(X; T\_l) \|\_F^2 + \mathcal{L}\_{\text{aux}} \big)$$

$D\_{\pm}$ uses straight-through estimator (STE) with a larger learning rate; $P$ uses standard gradient descent. at inference, $P^{-1}$ is absorbed into adjacent weights, so the transformation has zero runtime cost.

**compression ratios.** vector length $v$ and codebook size $c$ control the bit width. the ablation on LLaMA-2-7B at 1.11 bits shows:

| config | W-Bits | WikiText2 PPL | mean acc | quant time |
|--------|--------|---------------|----------|------------|
| v=4, c=9 | 1.11 | 39.97 | 36.52 | 43 min |
| v=16, c=7132 | 1.11 | 8.75 | 49.84 | 52 min |
| v=20, c=65536 | 1.11 | 6.06 | 61.84 | 66 min |

larger $v$ gives better quality but slower quantization. the sweet spot is $v = 128, c = 512$ for 0.8125 bits.

## training

PTQ method (no retraining of the LLM). the quantization pipeline is:

1. apply learnable transformation per block (optimize $T = D\_{\pm} P$ using calibration data)
2. binarize the transformed weights using ARB-LLM's naive procedure
3. run binary K-means codebook on the binarized sub-vectors
4. store codebook + indices

calibration uses WikiText2 validation set. the transformation optimization uses a per-block reconstruction objective with an auxiliary loss $\mathcal{L}\_{\text{aux}}$. quantization time for LLaMA-2-7B at $v = 128$ is approximately 66 minutes on a single GPU. no code is publicly available.

## evaluation

**perplexity (WikiText2), LLaMA family:**

| model | W-Bits | BiLLM | ARB-LLM | STBLLM | BTC-LLM |
|-------|--------|-------|---------|--------|---------|
| LLaMA-1-7B | 1.11 | 49.79 | 14.03 | - | **6.23** |
| LLaMA-1-7B | 0.80 | - | - | 15.03 | **6.72** |
| LLaMA-1-7B | 0.70 | - | - | 19.48 | **10.72** |
| LLaMA-2-7B | 1.11 | 32.31 | 16.44 | - | **6.06** |
| LLaMA-2-7B | 0.80 | - | - | 13.06 | **6.60** |
| LLaMA-2-13B | 0.80 | - | - | 11.67 | **5.83** |

BTC-LLM at 0.9 bits matches its own 1.11-bit performance (6.07 vs 6.06 PPL on LLaMA-2-7B), meaning the codebook captures redundancy almost perfectly. even at 0.7 bits, LLaMA-1-7B achieves 10.72 PPL vs 19.48 for STBLLM.

**zero-shot accuracy (7 tasks, average), 0.8 bits:**

| model | FP16 | STBLLM | BTC-LLM | delta vs FP16 |
|-------|------|--------|---------|---------------|
| LLaMA-1-13B | 63.80 | 55.83 | **60.82** | -3.1% |
| LLaMA-1-30B | 67.40 | 60.10 | **64.48** | -4.3% |
| LLaMA-2-13B | 65.00 | 53.85 | **61.91** | -4.8% |

BTC-LLM at 0.8 bits on LLaMA-2-13B loses only 3.1 percentage points on average, compared to STBLLM's 11.15 point loss.

**speedup.** BTC-LLM at 0.8 bits delivers 1.6$\times$ wall-clock speedup over FP16 on standard GPU hardware (no custom accelerator needed), thanks to LUT-GEMM replacing multiply-accumulate with table lookups.

**ablation on transformation components (LLaMA-2-7B):**

| config | WikiText2 PPL | mean acc |
|--------|---------------|----------|
| no transform | 9.23 | 49.54 |
| P only | 6.95 | 55.64 |
| $D\_{\pm}$ only | 7.33 | 56.48 |
| $D\_{\pm} + P$ (full) | **6.06** | **61.84** |

both components are necessary. the sign-flip matrix $D\_{\pm}$ alone contributes as much as the full affine $P$, suggesting that simple channel-wise sign alignment is a major source of quality recovery in binary quantization.

## reproduction guide

no public code available as of april 2026. to reproduce from the paper:

1. install the ARB-LLM quantization pipeline (it's open source) as the binary quantizer
2. implement the binary codebook with XOR+POPCNT K-means on packed int64 vectors
3. implement the learnable transformation: Kronecker-factored $P = P\_1 \otimes P\_2$ with STE for $D\_{\pm}$
4. calibrate on WikiText2 validation set, optimize per transformer block

estimated compute: single GPU (A100 or equivalent), ~1-2 hours for LLaMA-2-7B full quantization at 0.8 bits. the main cost is the per-block transformation optimization, which requires forward passes through each block.

key gotcha: the codebook quality depends heavily on the vector length $v$. too small (e.g., $v = 4$) and the centroids cannot capture meaningful patterns; too large (e.g., $v = 256$) and the number of unique patterns explodes, defeating compression. the paper's sweet spot is $v = 128$ with $c = 512$ for 0.8125 bits.

## notes

this paper is significant for the edge inference angle because it achieves sub-1-bit compression without sparse masks, meaning it runs efficiently on standard hardware (GPUs, CPUs) rather than requiring specialized sparse accelerators. the LUT-GEMM approach is particularly relevant for embedded deployment where lookup tables are cheap.

compared to the bonsai-1bit-8b approach (which uses a Q1\_0\_g128 format with FP16 group scaling), BTC-LLM goes further below 1 bit by exploiting the redundancy that binary codes naturally exhibit. the binary codebook idea is orthogonal to group quantization -- you could combine both for even better compression.

the $D\_{\pm}$ sign-flip matrix result is interesting mechanistically: it shows that much of the quantization error in binary methods comes from sign misalignment across channels, not magnitude errors. this connects to the hadamard rotation finding in our existing notes -- both are cheap transformations that disproportionately improve quantization quality. the difference is that $D\_{\pm}$ is learned per-model while hadamard is fixed.

the 22$\times$ memory savings at 0.7 bits (while maintaining usable perplexity) makes LLaMA-2-65B fit in ~4.3 GB, which is actually deployable on mobile devices. the 1.6$\times$ speedup over FP16 is modest but meaningful -- the real win is memory, not compute.
