## problem

The paper addresses a fundamental question in AI: **can a 1.3M parameter specialized model outperform general-purpose LLMs at real-time control?** The authors demonstrate this using DOOM (VizDoom's `defend_the_center` scenario) as a benchmark for real-time spatial reasoning under time pressure.

**Why this matters for tiny/embedded AI:** The rapid scaling of LLMs has created an assumption that bigger is always better. This work provides a concrete counterexample: a model with $92{,}000\times$ fewer parameters than Nemotron-120B achieves $14\times$ more frags. The implications are direct for edge deployment -- at $\sim$5MB (FP32), this model runs on a Raspberry Pi Zero 2W (512MB RAM, quad-core ARM Cortex-A53), no GPU or cloud required.

**Prior work on game-playing agents:**
- Deep RL approaches (Lample & Chaplot 2017; Dosovitskiy & Koltun 2017) trained DOOM agents from pixels using millions of frames
- GameNGen (Valevski et al. 2024) used diffusion models on 70M PPO agent frames to generate DOOM gameplay -- orders of magnitude more parameters
- Voyager (Wang et al. 2023) used GPT-4 as a Minecraft agent through code generation
- Waytowich et al. (2024) found multimodal LLMs (GPT-4V, Gemini) are incapable of zero-shot Atari gameplay
- Gallotta et al. (2024) identified real-time spatial games as fundamentally unsuited for LLM control

**Why small models matter:** The TinyML movement (Banbury et al. 2021) showed models under 1MB can run on microcontrollers. Belcak et al. (2025) argue small LMs are the future of agentic AI with 10--30$\times$ cost savings. This paper pushes that frontier into real-time game control -- a domain requiring sub-35ms decisions where LLMs need 0.6--13.3 seconds per frame.

## architecture

The model (SauerkrautLM-Doom-MultiVec) is a 5-layer ModernBERT encoder with three key modifications: hash embeddings, depth-aware token representations, and an attention pooling classification head.

**Input representation -- ASCII + Depth Encoding:**

Each DOOM frame $F \in \mathbb{R}^{H \times W \times 3}$ (640$\times$480 RGB, 921 KB) is converted to a 1,024-character ASCII string ($\sim$1 KB) -- a $900\times$ compression. The pipeline:
1. Grayscale conversion + downscale to $40\times 25$ via block averaging
2. Map each pixel to one of 10 brightness characters: `" .:-=+*#%@"` (dark to bright)
3. Newline separators produce exactly 1,024 tokens per frame

Depth buffer $D \in \mathbb{R}^{H \times W}$ is downscaled to $40\times 25$, normalized to $[0,1]$, and quantized into **16 discrete bins** ($\sim$40cm resolution at typical engagement distances). A 17th embedding handles special tokens ([CLS], [PAD]). Each token's final input embedding fuses both signals:

$$\mathbf{e}_{i} = \text{HashEmb}(\text{token}_{i}) + \text{DepthEmb}(\text{depth\_bin}_{i})$$

**ModernBERT-Hash Encoder (5 layers, H=128):**

| Property | Value |
|---|---|
| Hidden size | 128 |
| Attention heads | 4 (head dim = 32) |
| FFN intermediate size | 512 |
| Layers | 5 |
| Local attention window | 128 tokens |
| Global attention layers | 0, 3 (every 3rd layer) |
| Local attention layers | 1, 2, 4 |
| Position embeddings | RoPE (Rotary) |
| Flash Attention 2 | Native support |

**Hash Embeddings** replace the standard $V \times H = 128 \times 128 = 16{,}384$ parameter embedding table with a two-stage scheme:
1. Compact lookup table: $V \times P = 128 \times 16 = 2{,}048$ parameters (vocab is actually 75, so $75 \times 16 = 1{,}200$)
2. Linear projection: $P \times H = 16 \times 128 = 2{,}048$ parameters
3. LayerNorm on top

This reduces embedding parameters by **73%** (from 16,384 to 4,480) while maintaining representation quality. Drawn from Svenstrup et al. (2017) hash embeddings and Weinberger et al. (2009) feature hashing.

**Character-Level Tokenizer (75 tokens):**
- 10 brightness characters, 7 entity markers, 26 lowercase letters, 10 digits, 10 punctuation marks
- 6 action-specific special tokens ([ACT\_SHOOT], etc.)
- 5 standard special tokens ([PAD], [CLS], etc.)
- No subword merging -- strict one-to-one character-to-token mapping preserves 2D spatial coherence

Standard BPE tokenizers destroy ASCII art spatial structure (Luo et al. 2024, ASCIIBench), which is why a 1.3M model with 75 tokens outperforms 8B+ models with 100K+ vocabularies on spatial tasks.

**Attention Pooling Classification Head:**

The encoder outputs per-token embeddings $\mathbf{H} \in \mathbb{R}^{S \times 128}$ where $S \approx 1{,}024$. Learned attention pooling aggregates these:

$$\alpha_{i} = \frac{\exp(\mathbf{w}^{\top}\mathbf{h}_{i})}{\sum_{j}\exp(\mathbf{w}^{\top}\mathbf{h}_{j})}, \quad \mathbf{v} = \sum_{i}\alpha_{i}\mathbf{h}_{i}$$

where $\mathbf{w} \in \mathbb{R}^{128}$ is a learned query vector. The pooled representation is classified:

$$\mathbf{y} = \mathbf{W}_{c}\mathbf{v} + \mathbf{b}_{c} \quad \text{where } \mathbf{W}_{c} \in \mathbb{R}^{4 \times 128}$$

This provides the necessary information bottleneck (128-dim) while letting the encoder maintain rich per-token embeddings. Chosen over mean pooling (dilutes signal from game-relevant tokens) and CLS pooling (hard to learn from 31K samples).

**Architecture Evolution (3 phases):**
1. **ColBERT-style MaxSim** (abandoned): MaxSim scoring collapsed to single action; shared lowercase characters dominate the sum
2. **Learned Prototypes** (abandoned): 4 prototypes per action, both MaxSim and soft cross-attention collapsed; 1,024 $\times$ 128 = 131K input dimensions massively overparameterized for 4-class classification on 31K samples
3. **Attention Pooling** (adopted): Best classification accuracy (57.7%) and strongest gameplay

**Parameter Budget:**

| Component | Parameters | % of Total |
|---|---|---|
| Hash Embeddings (75 vocab, 16 proj) | 4,480 | 0.3% |
| Depth Embeddings (17 bins $\times$ 128) | 2,176 | 0.2% |
| Transformer Layers ($\times$ 5) | 1,312,000 | 99.1% |
| Attention Pool + Classifier | 644 | 0.05% |
| **Total** | **1,319,300** | **100%** |

Model size: $\sim$5MB (FP32). Loaded via HuggingFace Transformers with `trust_remote_code=True`.

## training

**Hardware:** Single NVIDIA RTX 6000 Ada GPU

**Human Demonstration Data:**
- Scenario: `defend_the_center` (enemies approach from all directions in circular arena)
- 4 recording sessions, 80+ episodes total, $\sim$2 hours of gameplay
- First author played using native DOOM keyboard controls (arrow keys + Ctrl for shooting)
- VizDoom spectator mode records both player inputs and full game state including depth buffer
- Total frames: **31,645** at frame\_skip=4 (one sample per 4 game tics)
- Per frame: ASCII text ($40\times 25$), depth bins (16-bin quantized per token), soft action scores (4-dim)

**Soft Labels:** When human presses multiple keys simultaneously (e.g., forward + shoot), both actions receive score 0.85; inactive actions receive 0.05 baseline. This provides richer supervision than one-hot labels -- the model learns that frames showing an enemy ahead warrant both shooting and advancing.

**Data Split:** Random 90/10 train/validation split: 28,480 train, 3,165 validation frames

**Training Configuration:**
- Optimizer: AdamW
- LR schedule: Cosine with 500 warmup steps
- Initial learning rate: $3 \times 10^{-4}$
- Batch size: 32
- Epochs: 50
- Mixed precision: bfloat16
- Checkpoint selection: Best validation accuracy
- No class resampling (natural distribution); soft labels mitigate imbalance implicitly

**Loss Function:** KL divergence between model's softmax output and soft teacher scores:

$$\mathcal{L} = \text{KL}\left(\text{softmax}(\mathbf{y}) \,\|\, \text{softmax}(\mathbf{s})\right)$$

where $\mathbf{s}$ is the 4-dimensional soft score from human actions. KL divergence chosen over cross-entropy because soft labels maintain non-zero loss even for well-classified examples (0.85/0.05 targets), improving training stability.

**Result:** 57.7% top-1 action prediction accuracy on validation set (substantially above 25% random baseline, but below 100% due to inherent ambiguity -- many frames are genuinely ambiguous).

**Composite Action Selection (inference):**
1. Classify frame $\to$ action probabilities
2. Select top action (e.g., `move_forward`)
3. Add `shoot` if $P(\text{shoot}) > 0.75 \times P(\text{top})$ (relative threshold)
4. Combine compatible actions (movement + rotation) if second action's probability $> 0.15$

This enables simultaneous actions like "move forward while turning left and shooting." 62% of the model's actions are composites. The model achieves a 74% episode survival rate (reaching timeout) compared to 0% for all reasoning LLMs.

## evaluation

**Main Benchmark** (frame\_skip=4, $\sim$114ms real-time per decision, `defend_the_center` scenario, 640$\times$480, HUD on, real-time pacing):

| Agent | Params | Episodes | Avg Surv. | Max Surv. | Total Frags | Latency |
|---|---|---|---|---|---|---|
| **SLM-Doom-MV-1.3M** | **1.3M** | **10** | **388** | **525** | **178** | **31ms** |
| GPT-4o-mini | proprietary | 10 | 104 | 228 | 0 | 646ms |
| Nemotron-120B | 120B | 5 | 88 | 104 | 3 | 8.9s |
| Qwen3.5-27B | 27B | 3 | 87 | 109 | 2 | 13.3s |
| Gemini Flash Lite | proprietary | 10 | 81 | 97 | 8 | 920ms |

**Key numbers:**
- $178$ frags total (17.8 per episode) vs $13$ frags combined for all LLMs = $14\times$ advantage
- Nemotron-120B has $92{,}300\times$ more parameters but only 3 frags
- GPT-4o-mini: **zero frags** across 10 episodes, purely evasive ($>60\%$ rotation actions)
- Our model latency: 31ms (within DOOM's 35 FPS native rate; below 100ms human reaction threshold)
- Nemotron latency: 8.9s ($287\times$ slower); Qwen3.5: 13.3s ($429\times$ slower)
- Even fastest LLM (GPT-4o-mini at 646ms) is $21\times$ slower

**Fair-Mode Benchmark** (frame\_skip=20, $\sim$571ms per decision, reduced pace to help LLMs):

| Agent | Params | Episodes | Avg Surv. | Max Surv. | Total Frags | Latency |
|---|---|---|---|---|---|---|
| **SLM-Doom-MV-1.3M** | **1.3M** | **10** | **33** | **46** | **10** | **29ms** |
| GPT-4o-mini | proprietary | 10 | 30 | 55 | 0 | 620ms |
| Gemini Flash Lite | proprietary | 10 | 16 | 18 | 5 | 1.1s |
| Qwen3.5-27B | 27B | 3 | 15 | 18 | 1 | 12.0s |
| Nemotron-120B | 120B | 5 | 14 | 20 | 0 | 26.7s |

Even with reduced pace, our model leads in both survival and frags. Gemini Flash Lite shows best LLM frag rate (5 frags), suggesting faster inference compensates for fewer parameters. Reasoning models remain too slow.

**Action Distribution:**
- Our model: $\sim$35% movement, 30% rotation, 35% shoot-containing (balanced)
- GPT-4o-mini: $>60\%$ rotation (pure evasion)
- Gemini Flash Lite: most diverse LLM distribution (correlates with highest frag count)

**Why LLMs Struggle (5 factors):**
1. **Latency**: enemies move 3--5m during LLM processing time
2. **BPE destroys spatial coherence**: merges character sequences into single tokens
3. **Spatial reasoning on text**: LLMs trained on verbal spatial relationships, not 2D character grids
4. **No temporal state**: each frame is a new prompt with no memory
5. **Defensive bias**: no gameplay objective in prompt $\to$ default to self-preservation (evasion)

## notes

**Implications for tiny model design:**

This paper is a compelling existence proof that architecture + data specialization beats brute-force scaling for real-time control. The key design insights:

1. **The right tokenizer matters more than model size.** A 75-token character-level vocabulary that preserves spatial one-to-one mapping outperforms 100K+ token BPE vocabularies on ASCII spatial reasoning. This directly connects to binh's work on LLMs on tiny devices -- tokenizer design is a critical lever for edge AI that's often overlooked.

2. **Depth as a learned modality is parameter-efficient.** 16 depth bins encoded as 2,176 learned embedding parameters provide the same spatial information that LLMs receive as 1,000 extra text tokens per frame. This is a 7:1 information-to-parameter advantage for the small model.

3. **Information bottleneck enables training stability.** The paper's failed experiments with ColBERT MaxSim and learned prototypes (both collapsed) vs successful attention pooling reveal that for small-data regimes (31K samples), you need a bottleneck between the rich encoder and the classification head. This is a general principle for tiny model design.

4. **Task architecture > general architecture.** ModernBERT's local+global attention pattern was specifically designed for the spatial locality of ASCII frames. The sliding window of 128 tokens matches the 40-column width of the ASCII representation, meaning local attention captures horizontal structure (walls, corridors) while global attention handles vertical relationships.

**Connections to binh's LLMs-on-tiny-devices work:**

- **Deployment target alignment:** The paper explicitly targets Raspberry Pi Zero 2W (512MB RAM, ARM Cortex-A53, no GPU) -- exactly the class of devices binh focuses on. At $\sim$5MB FP32 ($\sim$2.5MB int8), this model fits comfortably in microcontroller flash.
- **The "right-sized model" thesis:** The paper's conclusion -- "the future of AI is not exclusively large; it is *appropriately sized* for the task at hand" -- directly echoes binh's research philosophy.
- **Cost argument:** $15--$30 in API fees for LLM benchmarks vs free CPU inference indefinitely. For robotics, game AI, and autonomous systems requiring millions of inferences, this is decisive.
- **Latency argument:** 31ms vs 646ms--13.3s. Research on FPS games shows 35% accuracy drop at just 100ms latency (Claypool & Claypool 2006). All tested LLMs exceed this by 6--133$\times$. This extends to autonomous driving, industrial control, real-time bidding.
- **ONNX export path:** Future work targets sub-200ms inference on ARM Cortex-A53 via ONNX quantization -- a direct path binh could explore for deploying similar architectures on bare-metal microcontrollers.

**What makes this work vs LLMs -- a concise summary:**

The 1.3M model wins because it has: (1) a tokenizer that preserves 2D spatial structure, (2) depth as a fused learned embedding rather than appended text, (3) an architecture designed for the spatial locality of the task, (4) 31ms inference enabling real-time reactive play, and (5) training on task-specific data (31K human demos) rather than general text. The LLMs lose despite 92,000$\times$ more parameters because BPE destroys ASCII spatial coherence, their text reasoning cannot interpret 2D character grids, multi-second latency makes real-time play impossible, and they have no temporal state across frames. GPT-4o-mini's zero-frag, pure-evasion behavior is particularly telling -- without explicit gameplay objectives in the prompt, LLMs default to self-preservation rather than engagement.
