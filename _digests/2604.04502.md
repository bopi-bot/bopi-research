## problem

The paper investigates a central question: **how far can today's most advanced video generation models (specifically Google's Veo-3) support generalizable robotic manipulation without any task-specific demonstrations?**

Current VLA (Vision-Language-Action) models like $\pi\_{0.5}$, OpenVLA, and RT-2 are built by adapting pretrained VLMs to output robot actions. This adaptation process often degrades the VLM's generalization capability due to catastrophic forgetting—the action modality disrupts pretrained knowledge. Prior work such as Pi0, VPP (Video Prediction Policy), Vidar, and AnyPos has explored using video generation models for policy learning, but these rely on **open-source** video models that are significantly less capable than commercial frontier models (Veo, Sora). Additionally, video models are known to be inaccurate during contact-rich physical interactions, which limits their utility for dexterous manipulation.

The key gap: no prior work has systematically evaluated a **frontier commercial video model** (Veo-3) for zero-shot robotic manipulation, nor proposed a framework that combines its strong semantic planning with a specialized low-level executor.

## architecture

Veo-Act is a **hierarchical planning and control framework** with three core components:

### 1. Video Generation (Veo-3)
Given the initial observation image $I\_0$ and a task instruction prompt, Veo-3 generates a future video sequence:
$$I^{*}\_{0:n} = \{I^{*}\_1, I^{*}\_2, \ldots, I^{*}\_n\}$$
where $I^{*}\_k$ is the $k$-th synthesized frame. The prompt is a detailed template describing the robot, target object, container, and required actions. Only the **global-view camera** is used as input to Veo-3 (not the wrist camera).

### 2. Multi-Head Inverse Dynamics Model (IDM)
A multi-head IDM maps image transitions to robot actions **and** predicts an interaction gate for switching control:

$$(a\_t, G\_t) = \pi\_{\mathrm{IDM}}(I\_{t-1}, I\_t, s\_t)$$

where $s\_t$ is the 21-dimensional robot state, $a\_t$ is the predicted action, and $G\_t \in [0,1]$ is the gate value.

**Visual encoder:** DINOv3 ViT-B/16 (distilled), producing $768 \times 2 = 1536$-dimensional features from 2 consecutive frames.

**Action head (multi-head version):**
$$\text{Pose: } \text{Linear}(1536 \to 256) \to \text{ReLU} \to \text{Linear}(256 \to c\_{\mathrm{dim}})$$
$$\text{Grasp: } \text{Linear}(1536 \to 256) \to \text{ReLU} \to \text{Linear}(256 \to b\_{\mathrm{dim}})$$

**Single-head alternative:**
$$\text{Linear}(1536 \to \text{output\_dim})$$

**Loss function:**
$$\mathcal{L} = \lambda\_{\mathrm{act}} \cdot \mathcal{L}\_{\mathrm{act}}(a\_t, \hat{a}\_t) + \lambda\_{\mathrm{gate}} \cdot \mathcal{L}\_{\mathrm{gate}}(G\_t, \hat{g}\_t)$$

where $\mathcal{L}\_{\mathrm{act}}$ is a **weighted smooth L1 (Huber) loss** and $\mathcal{L}\_{\mathrm{gate}}$ is **Binary Cross Entropy**. The weighted Huber loss is:
$$d(x, \hat{x}) = \begin{cases} 0.5 \cdot w \cdot \frac{(x - \hat{x})^2}{\beta}, & |x - \hat{x}| < \beta \\ w \cdot |x - \hat{x}| - 0.5\beta, & \text{otherwise} \end{cases}$$
with $\beta = 0.1$ and per-dimension weights $w\_4 = w\_{11} = 2$, $w\_d = 1$ for all other $d \in \{0,\ldots,13\} \setminus \{4,11\}$.

### 3. Action Smoothing
A temporal smoother processes raw action chunks:
$$\bar{a}^{*}\_{0:n-1} = \mathrm{Smoother}(a^{*}\_{0:n-1})$$
Uses recursive extrema keypoint selection over $R$ rounds with minimum segment length $L\_{\mathrm{min}}$, centered moving-average inside keypoint intervals with window $w = 4$ and $w = 11$ for dimensions 4 and 11 respectively (other dimensions use $w = 1$), keypoint hold-stretching of $H$ extra steps, and a safety clamp on dimension $d\_c = 2$ with minimum bound $m = 0.13$.

### 4. Hierarchical Execution with Gated Switching
The system maintains an action queue $\mathcal{Q}$ and a gate history $\mathcal{H}\_G$. At each timestep:
- If gate $G\_t$ (from real observations via IDM gate head) is **stably above** threshold $\tau = 0.5$ for $K$ consecutive steps → switch to low-level VLA policy $\pi\_{\mathrm{VLA}}$
- If gate $G\_t$ is **stably below** $\tau$ → resume consuming the planned action queue from the next low-gate segment
- When switching back, the queue is **pruned** to skip high-gate segments (preventing re-entering the same interaction region)

Two simplified variants:
- **Pure IDM**: No switching; always pop from queue
- **Simultaneous Control**: Video plan controls arm pose; VLA always controls hand/gripper (no discrete switching)

### Ablation Variants
- **ResNet-50 backbone**: Input channels $3 \times 2 = 6$, feature dim $512 \times 4 = 2048$, same MLP head structure but $2048 \to 256 \to c\_{\mathrm{dim}} / b\_{\mathrm{dim}}$

- **Single-head IDM**: Only action head, no separate interaction detector

## evaluation

### Settings (all pick-and-place tasks with 7-DoF arm + 12-DoF dexterous hand)

**Simulation (30 trials per setting):**
1. Wrist-camera invisible — target outside wrist FOV
2. Similar-object distractors — e.g., {mango, potato}, {tomato, red cube}
3. Pass-by interaction — distractor along reaching path

**Real robot (13–19 trials per setting):**
1. Similar-object distractors (16 trials)
2. Pass-by interaction (13 trials)
3. Richer semantics — compositional instructions (19 trials)

**Metrics:** Instruction-following success (end-effector reaches target) and overall task success (object placed in container). 1-switch setting used for Veo-Act.

## reproduction guide

### Prerequisites
- Access to Veo-3 API (Google DeepMind)
- 7-DoF robot arm + 12-DoF dexterous hand with global-view and wrist cameras
- 4× NVIDIA Ampere 80GB GPUs for IDM training
- IsaacLab simulation environment (for sim data collection)

### Step-by-Step

1. **Environment setup:**
```bash
# Clone and install dependencies (no public repo available as of submission)
# Build IsaacLab simulation matching real-world setup
pip install torch torchvision
pip install timm  # for DINOv3
```

2. **Collect random-play data (simulation):**
   - 300k frame-pair samples with random motions + grasp/release
   - 100k additional pure random-motion samples
   - Record global-view images + 21-dim robot state per step
   - Label: grasp steps = 1, non-grasp = 0

3. **Collect random-play data (real world):**
   - 150k frame-pair samples on physical platform

4. **Apply STEM-OB augmentation:**
   - 50 denoising steps, 10 inversion steps on all RGB trajectories

5. **Train the multi-head IDM:**
```bash
# Training configuration
python train_idm.py \
  --backbone dinov3_vitb16 \
  --batch_size 8 \
  --gpus 4 \
  --iterations 85000 \
  --lr_backbone 5e-5 \
  --lr_rest 5e-4 \
  --optimizer adamw \
  --weight_decay 0.01 \
  --scheduler cosine \
  --warmup 8500 \
  --loss_act weighted_smooth_l1 \
  --loss_gate bce \
  --beta 0.1 \
  --dim_weights "4:2,11:2"
```
   - Expected training time: ~10 hours on 4× A100 80GB

6. **Train $\pi\_{0.5}$ low-level policy:**
```bash
# Using official pi0.5 implementation
python train_pi05.py \
  --batch_size 32 \
  --lr 2.5e-5 \
  --iterations 40000 \
  --action_space absolute \
  --normalize mean_std
```

7. **Run Veo-Act inference:**
   - Capture global-view image $I\_0$
   - Build video-generation prompt from task instruction
   - Query Veo-3 for future trajectory $I^{*}\_{0:n}$
   - Run IDM action head on $I^{*}\_{0:n}$ to get action chunk
   - Apply action smoother
   - Execute with gated switching: IDM gate head monitors real observations; switches to $\pi\_{0.5}$ when $G\_t > \tau = 0.5$ for $K$ consecutive steps

### Gotchas
- **Veo-3 API dependency**: The entire pipeline depends on Google's Veo-3, which is a closed commercial API—no open-source alternative tested.
- **No public code released**: As of submission, no code repository is available.
- **Video model sensitivity**: Minor changes in robot appearance, camera viewpoint, or scene layout produce substantially different video predictions.
- **1-switch evaluation**: All reported Veo-Act results use at most 1 switch from IDM to VLA; multi-switch is not evaluated.
- **Action smoother hyperparameters** ($w$, $H$, $S$, $L\_{\mathrm{min}}$) are task-specific and may need retuning for different robots/environments.
- **Gate threshold $\tau = 0.5$** and persistence window $K$ are sensitive—improper tuning causes premature or late switching.

## notes

### Key Takeaways
1. **Frontier video models already solve "semantic guidance"**: Veo-3 consistently generates approximately correct task-level trajectories even zero-shot, correctly identifying target objects from complex language instructions.
2. **The bottleneck is contact-rich interaction**: Pure IDM achieves up to 0.77 instruction-following but near-zero overall success. The video model's limitations manifest almost entirely during grasping/placement.
3. **Hierarchical decomposition works**: By delegating semantic planning to Veo-3 and dexterous interaction to $\pi\_{0.5}$, Veo-Act achieves a 2.8–3.2× improvement over the VLA baseline alone.
4. **VLM-to-VLA knowledge degradation is real**: The paper empirically demonstrates that adding an action head to a VLM degrades its generalization, while the video-prediction-plus-IDM paradigm preserves the video model's full native capabilities.

### Connections to Other Work
- **VPP (Hu et al., 2024)**: Veo-Act significantly outperforms VPP (a prior video-prediction policy) across all settings—VPP achieves 0.50 max instruction-following vs Veo-Act's 0.93 in similar-object distractors.
- **Cosmos (Kim et al., 2026)**: Also fine-tunes video models for robot control but uses open-source models rather than frontier commercial ones.
- **VideoVLA (Shen et al., 2025)**: Directly uses video generators as VLA policies; Veo-Act differs by keeping video model frozen and using IDM for translation.
- **Vidar (Feng et al., 2025), AnyPos (Tan et al., 2025), TC-IDM (Mi et al., 2026)**: All use open-source video models + IDM; Veo-Act demonstrates that frontier models provide substantially better semantic planning.
- **Ctrl-World (Guo et al., 2025)**: Identifies similar issues with video model inaccuracy during contact-rich interactions.

### Next-Step Ideas
1. **Multi-switch evaluation**: The current 1-switch constraint is limiting; allowing multiple switches could improve performance on longer-horizon tasks.
2. **Online video replanning**: Instead of generating the full trajectory upfront, replan with Veo-3 at each switch-back point using the latest observation.
3. **Trainable video model fine-tuning**: The paper hints at fine-tuning approaches but focuses on zero-shot; LoRA/adapter fine-tuning of an open-source video model conditioned on robot data could close the interaction gap.
4. **Learned smoother**: Replace the hand-designed keypoint smoother with a learned temporal refinement network.
5. **Broader task diversity**: Current evaluation is limited to pick-and-place; test on tool use, multi-step assembly, or bimanual tasks.
6. **Replace Veo-3 dependency**: Build an open-source reproducible pipeline using an open video model (e.g., Cosmos) as the planner to democratize the approach.
