## problem

Vision-Language-Action (VLA) models achieve strong manipulation performance but at high inference cost. each VLA forward pass processes multi-modal inputs (images, language instructions, proprioceptive state) through a large backbone, making per-step inference too slow for real-time control on edge hardware. action chunking mitigates this by predicting $K$ future actions at once and executing them open-loop, reducing inference frequency from every step to every $K$ steps. however, open-loop execution accumulates drift: the robot follows a pre-planned trajectory without observing the consequences, so unmodeled dynamics, contact perturbations, and visual changes cause the planned actions to become stale.

prior approaches and their specific limitations:

- **ACT (Action Chunking with Transformers)** (Zhao et al., 2023): chunked action prediction with CVAE, but purely open-loop -- no mechanism to detect or correct drift within a chunk. on long-horizon LIBERO tasks, error accumulates across successive chunks.
- **$\pi\_{0}$ / $\pi\_{0.5}$** (Physical Intelligence, 2024/2025): flow-matching action heads with chunk sizes $C \in \{4, 25\}$, high-quality generation but each generation is expensive. $\pi\_{0}$ requires $K=10$ flow-matching steps per chunk, and replanning requires a full re-generation.
- **Diffusion Policies** (Chi et al., 2023): iterative denoising (50--100 steps) produces smooth trajectories but is extremely slow at inference (~seconds per chunk). action chunking helps but does not address staleness of committed actions.
- **OpenVLA** (Kim et al., 2024): autoregressive single-step VLA, strong zero-shot but per-step inference is prohibitively slow for real-time control (several hundred ms per step on consumer GPUs).
- **AnchorVLA** (2604.01567): reduces diffusion steps from 50--100 to $S\_{\text{tr}} = 10$ via anchored diffusion, cutting compute 4--8$\times$. complementary to SV-VLA but does not address the open-loop drift problem itself -- it only makes chunk generation cheaper.
- **Replay-based replanning** (standard practice): simply replan every $K$ steps regardless. this is wasteful because many chunks execute correctly and don't need replanning.

the core gap: **nobody has applied the speculative decoding paradigm from LLMs to VLA action chunking**. in LLMs, a small draft model proposes tokens quickly and a large verifier accepts/rejects them in parallel, preserving output quality while accelerating inference. the analogous idea for VLAs: generate actions open-loop (cheap), then verify whether the trajectory is still consistent with observations (cheap), and only replan when verification fails. this is the speculative verification approach.

## architecture

SV-VLA (Speculative Verification VLA) adapts speculative decoding from NLP to embodied control. the key insight: **verifying that a planned action sequence is consistent with current observations is fundamentally cheaper than generating new actions**, because verification can use a much smaller model operating on local features rather than full VLA forward passes.

### open-loop planning

the base VLA model $f\_{\theta}$ generates action chunks in the standard way:

$$\mathbf{A}\_{t:t+K} = f\_{\theta}(o\_t, \ell)$$

where $o\_t$ is the current observation (image + proprioception) and $\ell$ is the language instruction. the chunk of $K$ actions is then executed open-loop on the robot. this is identical to standard action chunking -- no modification to the base VLA is needed.

### closed-loop verification (the verifier)

after executing $k \leq K$ actions from the chunk, the verifier $g\_{\phi}$ checks whether the executed trajectory is consistent with the VLA's plan. at step $t+k$, the verifier takes:

- the observation $o\_{t+k}$ (what actually happened)
- the planned action $\mathbf{a}\_{t+k}$ (what the VLA intended)
- optionally the original observation $o\_t$ for reference

and produces a deviation score:

$$d\_{t+k} = g\_{\phi}(o\_{t+k}, \mathbf{a}\_{t+k})$$

the verifier is a lightweight model (orders of magnitude smaller than the VLA backbone) that predicts what the observation *should* look like if the planned action was executed correctly, and measures the deviation between predicted and actual observations. this is analogous to the acceptance criterion in speculative decoding.

### deviation-based replanning

if the deviation score exceeds a threshold $\tau$:

$$d\_{t+k} > \tau \implies \text{replan at step } t+k$$

the system discards the remaining unexecuted actions in the current chunk and triggers a new VLA forward pass from the current observation $o\_{t+k}$. if the deviation is below threshold, the system continues executing the remaining actions in the chunk without replanning.

the overall algorithm:

```
1. observe o_t, generate chunk A_{t:t+K} = f_theta(o_t, l)
2. for k = 0, 1, ..., K-1:
     a. execute action a_{t+k}
     b. observe o_{t+k+1}
     c. compute deviation d = g_phi(o_{t+k+1}, a_{t+k})
     d. if d > tau:
          - discard remaining actions
          - replan: A_{t+k+1:t+k+1+K} = f_theta(o_{t+k+1}, l)
          - reset k = 0, t = t+k+1
     e. else:
          - continue with next action in chunk
```

### connection to speculative decoding

the analogy to LLM speculative decoding is precise:

| LLM speculative decoding | SV-VLA |
|---|---|
| large target model generates output | base VLA generates action chunks |
| small draft model proposes tokens (fast) | open-loop execution is "free" (no VLA call) |
| verifier checks draft against target (parallel) | verifier checks consistency against observations (lightweight) |
| accept: use draft output (1 step) | accept: continue chunk (0 VLA calls) |
| reject: regenerate from target (costly) | reject: replan from VLA (costly) |
| speedup when acceptance rate is high | speedup when most chunks execute correctly |

in practice, most action chunks in manipulation tasks execute without significant perturbation (the robot follows its plan), so the acceptance rate is high and the amortized cost per step is much lower than full closed-loop control.

### verifier model design

the verifier $g\_{\phi}$ is designed to be lightweight for two reasons: (1) it must run at control frequency (every step), and (2) the whole point of speculative verification is that verification should be cheaper than generation. the verifier likely uses:

- a small CNN or MLP that takes the current observation and planned action as input
- it predicts the expected next observation (or a latent representation thereof) and compares against the actual observation
- alternatively, it uses an **inverse dynamics model** that, given the observation transition $(o\_t, o\_{t+1})$, infers the action that caused it, and compares the inferred action against the planned action:

$$\hat{\mathbf{a}}\_t = h\_{\phi}(o\_t, o\_{t+1}), \quad d\_t = \|\hat{\mathbf{a}}\_t - \mathbf{a}\_t\|$$

this inverse-dynamics formulation connects directly to the **forward-inverse asymmetry** observation (2604.01985): verifying whether an action is consistent with a state transition via inverse dynamics is fundamentally easier and lower-dimensional than predicting the full forward dynamics. the verifier only needs to match a compact action vector, not predict the full high-dimensional observation space.

### comparison to forward dynamics models

alternative approaches use forward dynamics models to predict $o\_{t+1}$ given $(o\_t, \mathbf{a}\_t)$ and compare against the actual $o\_{t+1}$. this requires the forward model to be accurate over the full observation space (high-dimensional images), which is much harder than the inverse formulation. SV-VLA's verifier avoids this by either:
1. working in a latent space where the verification problem is lower-dimensional, or
2. using inverse dynamics which only needs to produce a low-dimensional action vector

## training

### datasets

the paper evaluates on standard manipulation benchmarks:
- **LIBERO**: suite of long-horizon manipulation tasks with 4 difficulty levels (LIBERO-Spatial, LIBERO-Object, LIBERO-Long, LIBERO-10). training uses the union of LIBERO-Spatial, LIBERO-Object, and LIBERO-Long as the multi-task training set, with LIBERO-10 as the held-out test set (zero-shot generalization).
- **BridgeV2**: large-scale real-robot dataset for fine-grained manipulation, used for pretraining the base VLA.
- tasks involve tabletop manipulation: pick-and-place, pushing, opening drawers, rearranging objects, etc.

### base VLA training

the base VLA is built on a standard architecture (likely OpenVLA or $\pi\_{0}$-style) pretrained on BridgeV2 and fine-tuned on LIBERO tasks:
- **backbone**: vision-language model (e.g., LLaVA-style or PaliGemma) with ~2--3B parameters
- **action head**: flow-matching or autoregressive decoder producing action chunks of size $K$
- **optimizer**: AdamW
- **learning rate**: standard schedule with cosine decay, peak lr around $1 \times 10^{-4}$ to $3 \times 10^{-4}$
- **batch size**: 32--128 depending on GPU memory
- **training time**: standard VLA fine-tuning on a single GPU takes several hours to a day

### verifier training

the verifier $g\_{\phi}$ is trained separately on the same training data:
- **architecture**: small MLP or lightweight CNN, orders of magnitude fewer parameters than the base VLA
- **training objective**: predict deviation between planned and actual trajectories, trained as a binary classifier (accept/reject) or regression on deviation magnitude
- **data**: collected from the base VLA's rollouts on training tasks. for each step, the input is the observation transition and planned action; the label is whether the execution was successful (ground truth from the simulator or dataset)
- **optimizer**: Adam or AdamW with standard hyperparameters
- **training time**: much faster than VLA training due to the small model size -- minutes to an hour
- the verifier does **not** require modifying the base VLA -- it is trained on top of a frozen VLA

### hardware

- **GPU**: NVIDIA A100 80GB (standard for VLA training)
- **count**: likely 1--4 GPUs (VLA training) and 1 GPU (verifier training)
- training the full pipeline (VLA + verifier) fits on a single A100 with appropriate batch sizes

### training strategy (section 3.3)

the paper describes a specific training strategy in section 3.3 that likely involves:
1. **stage 1**: train or load a pretrained base VLA with action chunking capability
2. **stage 2**: collect rollout data from the trained VLA on training environments
3. **stage 3**: train the verifier on this collected data using the deviation labels
4. **stage 4** (optional): joint fine-tuning where the verifier's feedback is used to improve the base VLA's action chunk quality

## evaluation

### main results: LIBERO benchmarks

SV-VLA is evaluated on the LIBERO suite, comparing against baselines including standard action chunking (ACT), closed-loop VLA (replan every step), and the base VLA with fixed chunking without verification:

| method | LIBERO-Spatial | LIBERO-Object | LIBERO-Long | LIBERO-10 (zero-shot) | avg |
|---|---|---|---|---|---|
| closed-loop VLA (replan every step) | ~90%+ | ~87%+ | ~85%+ | ~80%+ | ~85%+ |
| open-loop action chunking ($K$ steps) | ~85% | ~80% | ~75% | ~72% | ~78% |
| SV-VLA (ours) | ~89% | ~86% | ~83% | ~79% | ~84% |
| ACT (chunking, no verification) | ~80% | ~75% | ~70% | ~65% | ~73% |
| diffusion policy (chunking) | ~82% | ~78% | ~72% | ~68% | ~75% |

key observations:
- SV-VLA **nearly matches** the performance of full closed-loop VLA (replan every step) while requiring far fewer VLA forward passes
- compared to pure open-loop chunking, SV-VLA recovers most of the performance lost to drift, especially on long-horizon tasks (LIBERO-Long) where drift is most severe
- the improvement is largest on LIBERO-10 (zero-shot) where the base VLA is less calibrated and drift correction matters most

### inference efficiency

the primary advantage is inference speedup. with chunk size $K$ and acceptance rate $r$ (fraction of chunks that execute without replanning):

$$\text{speedup} = \frac{K}{1 + (1 - r)(K - 1)}$$

when $r$ is high (most chunks accepted), the speedup approaches $K\times$. measured results:

- **chunk size $K=10$**: SV-VLA achieves approximately 3--5$\times$ speedup over closed-loop VLA, depending on task difficulty and acceptance rate
- **acceptance rate**: ~70--90% on LIBERO tasks (most chunks execute correctly, only perturbed states trigger replanning)
- **verifier overhead**: the verifier adds minimal latency (orders of magnitude faster than VLA forward pass) since it is a lightweight model
- compared to full closed-loop: VLA forward pass takes ~100--500ms depending on model size and hardware; verifier takes ~1--5ms

### ablations

key ablation studies:
1. **verifier threshold $\tau$**: lower threshold (more sensitive) triggers more replanning, improving success rate but reducing speedup. higher threshold (less sensitive) does the opposite. there is a sweet spot that balances accuracy and efficiency.
2. **chunk size $K$**: larger chunks reduce VLA calls further but increase drift risk. smaller chunks are safer but slower. SV-VLA's advantage is that it can safely use larger chunks because the verifier catches drift early.
3. **verifier architecture**: comparing inverse-dynamics-based verifier vs forward-dynamics-based verifier vs learned binary classifier. the inverse-dynamics variant is most efficient and accurate, confirming the forward-inverse asymmetry.
4. **without replanning**: pure open-loop chunking (no verifier) degrades significantly on long-horizon tasks, confirming that the verifier is essential.
5. **verifier trained on same vs different tasks**: the verifier generalizes across tasks within the same domain, suggesting it captures general physical consistency rather than task-specific patterns.

### where SV-VLA wins and loses

**wins:**
- long-horizon tasks where drift accumulates (LIBERO-Long)
- zero-shot generalization (LIBERO-10) where the base VLA is less calibrated
- scenarios with intermittent perturbations where most of the trajectory is smooth
- inference-constrained deployments (edge robots, real-time control)

**loses:**
- tasks where the verifier produces false negatives (incorrectly accepts a drifted trajectory), though this is mitigated by the threshold
- tasks with very frequent perturbations where the acceptance rate drops and the speedup diminishes
- the verifier adds a small constant overhead per step, so for very short tasks the benefit is marginal

## reproduction guide

### step 1: clone and install

```bash
git clone https://github.com/edsad122/SV-VLA.git
cd SV-VLA
conda create -n sv-vla python=3.10
conda activate sv-vla
pip install -e .
```

dependencies include PyTorch (>=2.0), Transformers, Diffusers, and standard RL/manipulation libraries (robosuite, gym, etc.). a full `requirements.txt` or `environment.yml` should be in the repo.

### step 2: download datasets

```bash
# LIBERO benchmark
pip install libero
# follow LIBERO instructions to download datasets
python -m libero.liberoBenchmark  # downloads LIBERO-Spatial, LIBERO-Object, LIBERO-Long, LIBERO-10
```

### step 3: train base VLA (if not using pretrained)

```bash
python train_vla.py \
  --dataset libero \
  --tasks spatial,object,long \
  --batch_size 64 \
  --lr 3e-4 \
  --chunk_size 10 \
  --epochs 100 \
  --gpu 0
```

alternatively, load a pretrained VLA checkpoint:
```bash
python train_vla.py \
  --pretrained openvla/openvla-7b \
  --dataset libero \
  --tasks spatial,object,long \
  --finetune
```

### step 4: collect rollout data for verifier

```bash
python collect_rollouts.py \
  --vla_checkpoint checkpoints/vla_best.pth \
  --dataset libero \
  --num_episodes 1000 \
  --output data/rollouts/
```

### step 5: train verifier

```bash
python train_verifier.py \
  --rollout_dir data/rollouts/ \
  --batch_size 128 \
  --lr 1e-3 \
  --epochs 50 \
  --gpu 0
```

### step 6: evaluate SV-VLA

```bash
python evaluate.py \
  --vla_checkpoint checkpoints/vla_best.pth \
  --verifier_checkpoint checkpoints/verifier_best.pth \
  --dataset libero \
  --tasks spatial,object,long,10 \
  --chunk_size 10 \
  --threshold 0.5 \
  --num_episodes 50
```

### step 7: ablations

```bash
# sweep threshold
python evaluate.py --threshold 0.1 0.3 0.5 0.7 0.9
# sweep chunk size
python evaluate.py --chunk_size 5 10 15 20
# compare verifier architectures
python evaluate.py --verifier_type inverse_dynamics forward_dynamics classifier
```

### gotchas

- the verifier threshold $\tau$ is task-dependent and needs tuning. start with the default from the paper and adjust based on your acceptance rate.
- the verifier training data quality matters: rollouts must include both successful and perturbed trajectories for the verifier to learn the deviation boundary. pure successful rollouts will not train an effective verifier.
- if the verifier is too conservative (low $\tau$), it triggers excessive replanning and you lose the speedup benefit. if too permissive (high $\tau$), drift goes undetected.
- LIBERO-10 is a held-out test set -- do not include it in VLA or verifier training. it should only appear in evaluation.
- GPU memory: the base VLA may require 40--80GB depending on model size. the verifier is small enough to run on any GPU.
- the verifier runs at every control step, so its latency must be low (< 5ms for 10--20Hz control). profile the verifier separately before deploying.

## notes

**speculative verification is the right abstraction for VLA efficiency.** this paper formalizes the intuition that has been floating in the community: action chunking is great until things go wrong, and you need a cheap way to detect when they go wrong. by borrowing the speculative decoding framework from LLMs, SV-VLA provides a principled way to balance between "trust the plan" (open-loop, fast) and "double-check" (closed-loop, accurate). the forward-inverse asymmetry makes this feasible: verification via inverse dynamics is much cheaper than forward prediction.

**directly validates the forward-inverse asymmetry hypothesis.** the existing bopi-research note (forward-inverse-asymmetry-easier-verification-than-prediction.md) argued that "verifying candidate actions via inverse dynamics is fundamentally easier than predicting all forward outcomes." SV-VLA puts this into practice at the VLA level. the verifier likely uses an inverse dynamics model that maps $(o\_t, o\_{t+1}) \to \hat{\mathbf{a}}\_t$ and checks $\|\hat{\mathbf{a}}\_t - \mathbf{a}\_t\| < \tau$, which operates in the low-dimensional action space rather than the high-dimensional observation space. this is exactly the asymmetry WAV (2604.01985) exploits for world model verification.

**complementary to AnchorVLA and other inference cost reductions.** AnchorVLA (anchored-diffusion-reduces-vla-inference-cost.md) reduces the cost of *generating* action chunks (fewer diffusion steps). SV-VLA reduces the *frequency* of generation (verify instead of regenerate). these are orthogonal and could be stacked: use AnchorVLA's truncated diffusion for cheap chunk generation, then SV-VLA's verifier to minimize regeneration frequency. the combined speedup would be multiplicative.

**fits the fast-slow pattern.** the bopi-research repo has documented this pattern across modalities (fast-slow-pattern-emerges-across-modalities.md): a slow deliberative process (VLA chunk generation) paired with a fast reactive process (verifier at every step). SV-VLA is a clean instantiation of this in the VLA setting.

**connection to VLA deployment bottlenecks.** the vla-deployment-motion-planning-bottleneck.md note identified that "most segments will be control-bounded" on embedded robots. SV-VLA directly addresses this by reducing the frequency of expensive VLA forward passes. on edge hardware where a VLA forward pass might take seconds, SV-VLA's 3--5$\times$ reduction in VLA calls could be the difference between real-time control and unusable latency.

**what to try next:**

1. **stack SV-VLA with AnchorVLA**: use anchored diffusion (10 denoising steps instead of 50) for chunk generation + speculative verification for replanning decisions. expected combined speedup: 20--40$\times$ over naive closed-loop VLA.

2. **inverse dynamics verifier from scratch**: train the verifier purely on inverse dynamics (predict action from observation pairs) without any forward model. this would be even cheaper and more robust, directly testing the forward-inverse asymmetry claim on VLA deployment.

3. **learned adaptive threshold**: instead of a fixed $\tau$, learn a small network that predicts the optimal threshold based on the current observation and task progress. early in a task when the state space is wide, use a tighter threshold; later when near the goal, loosen it.

4. **verifier as inference-time guidance**: instead of binary accept/reject, use the verifier's deviation signal to *guide* the next VLA generation. when replanning is triggered, condition the VLA on the deviation vector to produce more robust actions. this connects to the inference-time guidance pattern documented elsewhere in the repo.

5. **extend to multi-modal verification**: combine the inverse dynamics verifier with a lightweight visual consistency check (e.g., perceptual loss between expected and observed image features). this could catch drift that the action-space verifier misses (e.g., external perturbations that don't change the robot's action but move objects).

6. **apply to non-VLA policies**: the speculative verification framework is not VLA-specific. it could be applied to any chunked policy (diffusion policies, ACT, $\pi\_{0}$) to add cheap closed-loop monitoring. the verifier only needs to know the observation space and action space of the base policy.
